# Hosting CipherChat

This app should be hosted as one Node web service for the first real-time version. The Express server serves the built React app, the `/api` routes, and Socket.IO from the same public URL.

## Recommended Free/Freemium Stack

- App host: [Render web service](https://render.com/docs/web-services).
- Database: [MongoDB Atlas free cluster](https://www.mongodb.com/docs/atlas/reference/free-shared-limitations/).
- File storage: [Cloudinary free plan](https://cloudinary.com/pricing).

This keeps real-time messaging simple because the browser, API, and Socket.IO all use the same domain. It also avoids storing uploaded files on the server filesystem, which is not durable on many free hosts.

## 1. Create MongoDB Atlas

1. Create a MongoDB Atlas account.
2. Create a free cluster.
3. Create a database user with a strong password.
4. In Network Access, allow your app to connect. For a quick MVP, `0.0.0.0/0` is easiest. For a serious private app, restrict access when your host gives you stable outbound IPs.
5. Copy the driver connection string and replace `<password>` plus the database name.

Use it as:

```env
MONGO_URI=mongodb+srv://username:password@cluster.mongodb.net/private_e2ee_messenger?retryWrites=true&w=majority
```

## 2. Create Cloudinary

1. Create a Cloudinary account.
2. Open the dashboard and copy:
   - Cloud name
   - API key
   - API secret
3. The app stores message attachments as encrypted blobs. Cloudinary cannot read those message files because encryption happens in the browser before upload.
4. Profile pictures are normal public images because other users need to see them.

## 3. Push To GitHub

Render works best by connecting to a GitHub repository.

```bash
git add .
git commit -m "Prepare production deployment"
git push
```

## 4. Deploy On Render

Recommended: use the included `render.yaml` Blueprint.

1. In Render, choose **New** then **Blueprint**.
2. Connect your GitHub repo.
3. Render will read `render.yaml`.
4. When Render asks for secret values, enter:
   - `MONGO_URI`
   - `CLOUDINARY_CLOUD_NAME`
   - `CLOUDINARY_API_KEY`
   - `CLOUDINARY_API_SECRET`
5. `JWT_SECRET` is generated by Render from `render.yaml`.
6. Deploy.

Manual Render setup also works:

```text
Runtime: Node
Build command: npm ci && npm run build
Start command: npm start
Health check path: /api/health
```

Set these environment variables:

```env
NODE_ENV=production
HOST=0.0.0.0
SERVE_CLIENT_DIST=true
CLIENT_DIST_PATH=client/dist
STORAGE_DRIVER=cloudinary
CLOUDINARY_FOLDER=private-e2ee-messenger
MAX_ATTACHMENT_MB=25
MONGO_URI=your-atlas-connection-string
JWT_SECRET=generate-a-long-random-secret
CLOUDINARY_CLOUD_NAME=your-cloud-name
CLOUDINARY_API_KEY=your-api-key
CLOUDINARY_API_SECRET=your-api-secret
```

Leave `CLIENT_ORIGIN` empty when React and the API are on the same Render service.

## 5. First Live Test

1. Open your Render URL.
2. Register two users in two different browsers or devices.
3. Start a conversation.
4. Share the conversation key between both browsers.
5. Send a text message and an image.
6. Check `/api/health`; it should return `{"ok":true,"storage":"cloudinary"}`.

## Important Real-Time Notes

- Render web services support inbound WebSocket connections, which is why this app can run Socket.IO there.
- Render Free can spin down after 15 minutes without inbound traffic, so the first visit after inactivity may take about a minute and realtime sockets will reconnect.
- For truly always-on chat, upgrade the web service to a paid always-on instance later.
- Browser notifications in this project work while the app tab is open and connected. Notifications when the browser is fully closed need a Service Worker plus Web Push, which is a separate production feature.
- The E2EE model still stores conversation keys in each browser's local storage. Treat this as a private MVP, not a full Signal-grade protocol.
